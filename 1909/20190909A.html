【归纳】React 组件通信与 Redux

广宇 2019-9-9 Redux React

这篇博客本来是一月十五号就要写完的，那时应该已经回到遂川去了，在家备战春招。虽然那个时候想着要归纳一下 Redux 的知识，后面还是因为打瞌睡去睡觉去了，而且对这方面的知识也是一知半解，最终还是没有归纳出来。而现在 Redux 的内容也没有搞清楚，差不多也忘干净了。时间已经过去半年，也是时候补充回来这部分知识了。

回来这几天就水了一篇博客，就是一直在看 React 毕设课程，所以自己心里想写也有点不知道写什么。Redux 其实在毕设开始时课程中就讲过，已经很多天了。但是今天在学习的过程中，感觉对 Redux 的相关内容有点生疏了，正好这节课之后老师也补充了一节 React-Redux 的内容，真是瞌睡来了送枕头。本来想直接看掉，但是现在我确实有点困，所以现在就干脆做一些不容易打瞌睡的事情，干脆就把之前几次学的 Redux 的相关知识归纳一下。

# 一、React 的组件通信

# 1、父子组件使用 props 进行通信

在此前的博客中已经对 React 的一些基础知识做了介绍。React 中有类组件和函数组件，组件接受只读的props作为「参数」，渲染出相应的内容。此外，类组件可以有自己的状态 state，状态则由组件内部进行控制。而是用 React 搭建的页面，就是由一个个组件结合而成的。

但是在页面中编写的组件并不是独立存在的，相互之间需要进行通信。举一个常见的例子，点击按钮换出弹窗，实际上就是组件通信。可见组件之间的通信对组件开发是至关重要的。那么组件之间要如何通信呢？看一个简单的例子：

function Father(props) {
  function info1(info){
    console.log(info);
  };
  const info2 = '父组件传递给子组件的信息';
  return (
    <Son onClick={info} show={info2} />
  );
}

function Son(props){
  return (
    <div onClick={() => props.onCLick('子组件给父组件的信息')}>
      {props.show}
    </div>
  );
}

从上面的例子可以看出，父子组件之间可以通过props互相通信，父组件直接把需要传递的内容放到props中。如果父组件将其内部的某个函数作为props传递给子组件，子组件就可以通过调用这个函数从而达到把信息传递给父组件的效果。同理，如果是多层的父子关系，一样可以通过props，像接力一样达成通信的效果。

不过使用这种方法弊端十足。且不谈多层父子组件之间的叠加关系，如果是两个没有关系的独立组件，要想使用props就行不通了。尽管如此，在父子组件之间使用props通信还是非常高效的，平时工作中也会经常用到。

# 2、使用 EventHub 进行通信

EventHub 其实是早就学过的，可是之前并没有写博客记录下来。可能是当时觉得这部分内容有些难，就暂且没有归纳，久而久之竟然也记不太清楚了。这次一并把 EventHub 搞清楚。

EventHub 从字面是来看的话，就是「事件中心」的意思，要想明白「事件中心」是怎样一回事，就要了解「发布-订阅」模式。「发布-订阅」模式也被称为观察者模式，百度百科这样说：

观察者模式（有时又被称为模型（Model）-视图（View）模式、源-收听者(Listener)模式或从属者模式）是软件设计模式的一种。在此种模式中，一个目标物件管理所有相依于它的观察者物件，并且在它本身的状态改变时主动发出通知。这通常透过呼叫各观察者所提供的方法来实现。此种模式通常被用来实现事件处理系统。

要想将它用到组件的通信中，大概可以这么理解，发送消息的组件需要「发布」事件，接受消息的组件则需要「订阅」事件。负责处理事件的对象就可以称之为事件中心，也就是 EventHub。下面来看一个例子，两个无关的组件通过 EventHub 完成了通信：

var eventHub = {
  fnLists: [],
  trigger: (eventName, data) => {
    const fnList = this.fnLists[eventName];
    if (!fnList) {
      return null;
    }
    for (let i = 0; i < fnList.length; i++) {
      fnList[i](data);
    }
  },
  on: (eventName,fn) => {
    if (!this.fnLists[eventName]) {
      this.fnLists[eventName] = [];
    }
    this.fnLists[eventName].push(fn);
  }
};

function Comp1() {
  function info1(info){
    eventHub.trigger('我要发消息', info);
  };
  const info = '组件需要传递的信息';
  return (
    <div onClick={() => info1(info)}>
      点我发送消息
    </div>
  );
}

class Comp2 extends React.Component {
  constructor(){
    super();
    this.state = {
      info: ''
    };
    eventHub.on('我要发消息', (info) => {
      console.log('我收到消息了');
      this.setState({
        info
      });
    })
  }
  render() {
    return (
      <div>
        我收到的消息是：「{this.state.info}」
      </div>
    );
  }
}

这个例子看起来有点繁琐，需要理解的重点主要在 EventHub 上。在这个例子中，订阅事件就是在fnLists对象中，往以事件名对应的数组中推入相应的操作函数；发布事件则是事件名对应的数组中所有的回调函数都调用一遍，如果没有则说明没有组件订阅这个事件。感觉和订报纸有些相似，首先我要订阅相应的报纸（事件名），留下我的地址（回调函数），然后报纸发布时才会往地址送报（调用回调函数）。

通过这种方法，就可以实现组件间的通信。从上面的例子可以看出，EventHub 中所有组件必须通过事件沟通，而且 EventHub 必须放在顶层，才能保证所有的组件能够访问到。EventHub 看起来已经能满足组件通信的需求，为什么还要用其他的方法呢？与 EventHub 具有同样使命的 Redux 又是怎么来实现组件通信的呢？

# 二、Redux 与 React 组件通信

# 1、使用 Redux

什么是 Redux？随着 JavaScript 单页应用开发日趋复杂，JavaScript 需要管理比任何时候都要多的 state （状态），而 Redux 正好给提供了一种可预测化的状态管理方案。Redux 是 JavaScript 的一种状态容器，如果要将其与已知的一些概念类比，感觉它和 EventHub 比较相似。EventHub 比较随便，Redux 相当于将 EventHub 的各种行为进行约束了，虽然显得可能更加麻烦，但也更加规范了。

至于 Redux 是如何做这个规范的，一时半会还不好讲清楚。直接来看一个例子，如果刚才的例子的组件通信改为用 Redux 实现，大概是这个样子：

const store = Redux.createStore(reducers); // 创建store
const reducers = (state, action) => { // 管理更新store
  switch (action.type) {
    case '我要发消息':
      return {
        info: action.payload
      };
    default:
      return state;
  }
};

function Comp1({ store }) {
  function info1(info){
    store.dispatch({ type: '我要发消息', payload: info });
  };
  const info = '组件需要传递的信息';
  return (
    <div onClick={() => info1(info)}>
      点我发送消息
    </div>
  );
};

class Comp2 extends React.Component {
  constructor(){
    super();
    this.state = {
      store: this.props.store.getState()
    };
  }
  render() {
    return (
      <div>
        我收到的消息是：「{this.state.store.info}」
      </div>
    );
  }
};

const App = () => (
  <div>
    <Comp1 store={store} />
    <Comp2 store={store} />
  </div>
);

const render = () => {
  ReactDOM.render(<App />, document.querySelector('#app'));
}

render()
store.subscribe(render);

# 2、Redux 的基本组成

接下来就来了解一下 Redux 的基本组成。Redux 主要有以下几个组成部分：Action（动作）、reducer（处理器）、Store（状态管理）。上面的例子中都有涉及到，这里展开讲解。

# 1、Store

从之前的例子可以看出，Redux 中的 Store 起到了代替 EventHub 中事件中心的作用。Redux 应用只有一个单一的 Store。Store 用于管理整个应用的 state（状态）与 action（行为）。Store 具有以下职责：

  • 维持应用的 state；
  • 提供 getState() 方法获取 state；
  • 提供 dispatch(action) 方法更新 state；
  • 通过 subscribe(listener) 注册监听器;
  • 通过 subscribe(listener) 返回的函数注销监听器。

# 2、Reducer

Reducers 指定了应用状态的变化如何响应 actions 并发送到 store 的。Reducer 是一个纯函数，它接收两个参数，Store 之前的 state（状态）与需要执行的 action（行为），并返回新的 state。Reducer 用于描述 Action 如何更新 Store。需要谨记 reducer 一定要保持纯净。**只要传入参数相同，返回计算得到的下一个 state 就一定相同。**没有特殊情况、没有副作用，没有 API 请求、没有变量修改，单纯执行计算。

永远不要在 reducer 里做这些操作：

  • 修改传入参数；
  • 执行有副作用的操作，如 API 请求和路由跳转；
  • 调用非纯函数，如 Date.now() 或 Math.random()。

# 3、Action

Redux 有自己的文档，文档中的原话这样描述 Action，看起来有点高深：

Action 是把数据从应用传到 store 的有效载荷，它是 store 数据的唯一来源。

前面讲了，store 就是状态管理，把数据从应用传到 store 就相当于更新 store 中的状态。实际上这里的 Action 就相当于 EventHub 中的 event，这样说就比较好理解了，它实际上就是状态变更的发起行为。Redux 中的 Action 规定了格式，在创建 Action 实例时需要遵照这些格式。

Action 本质上是 JavaScript 普通对象。约定，Action 内必须使用一个字符串类型的 type 字段来表示将要执行的动作，起到了标记的 Action 的作用。除了 type 字段外，action 对象的结构完全由你自己决定。Flux Standard Action（FSA 规范）这样建议构造 Action：

  1. Action 是一个普通 JavaScript 对象，必须具有 type 属性
  2. Action 中只能具有这四种属性：type（动作名）、payload（参数）、error（是否错误）、meta（备注）

# 3、React-Redux

Redux 虽然规范化了 EventHub，但实际上也存在问题。因为之前的例子都比较简单，所以问题不是特别突出。这里的 Store 也好，EventHub 也好，都必须存在于应用的顶层。要想让所有组件都能访问到事件中心，还需要父级组件使用props将其传递给子孙组件，这样并不方便。因此，为了在 React 更好地使用 Redux，就有了 React-Redux 这个库，接下来就来简单地看看 React-Redux 究竟好在哪里。

使用 Provider 组件，可以让应用中的所有组件都能拿到 Store。例如：

const rootElement = document.getElementById("root");
ReactDOM.render(
  <Provider store={store}>
    <div>
      <Comp1 />
      <Comp2 />
    </div>
  </Provider>,
  rootElement
);

在 Provider 的子组件中，通过props就可以操作 Store 了，不过 React-Redux 提供了更加精细的功能。例如，在子组件Comp1中，需要通过高阶函数connect对组件进行封装。connect接受两个参数，mapStateToProps用于从 Store 中取出需要操作的 state，并将其映射到组件的props中；mapDispatchToProps则用于定义组件中需要用到的 Action，并同样将其映射到props：

function Comp1({ info, comeon }) {
  const info = '组件需要传递的信息';
  return (
    <div onClick={comeon}>
      目前的信息是「{info}」，点我更新
    </div>
  );
};

function mapStateToProps(state){
  return {
    info: state.info
  }
}

function mapDispatchToProps(dispatch) {
  return {
    comeon: ()=> dispatch({type:'我要发消息', payload: '我要发的信息'})
  }
}

export default connect(mapStateToProps,mapDispatchToProps)(Comp1);
