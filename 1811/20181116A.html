【归纳】虚拟DOM

广宇 2018-11-17 JavaScript

今天系统课学习了关于Vue与axios的知识，除此之外还有一些MVC相关的内容。所以本来今天是想写一篇MVC的博客，但是感觉MVC的内容比较多，而且理论知识比较多，所以过几天有空整理的时候再来写MVC的内容。

在我翻阅笔记的过程中，我看到我虚拟DOM这一节笔记做得比较乱，而且现在我感觉我关于虚拟DOM的内容也忘记不少，所以今天晚上索性把这一节的笔记整理一下。

# 什么是虚拟DOM

虚拟DOM又是一个在我吴天喜之前从来没有听过的一个名词。所以这次博客可能又要花上很多时间用来查资料了吧。就先从虚拟DOM到底是什么，它的作用以及它与DOM的关系这几点入手，最终达到把这个知识点弄清楚的效果。一个明明白白的关于虚拟DOM的具体定义，我并没有找到。所以只能自己结合一些其他人的见解再结合我的笔记进行归纳了。

虚拟DOM（virtual dom），是vue和react的核心。虚拟DOM概念随着react的诞生而诞生，由facebook提出，其卓越的性能很快得到广大开发者的认可；继react之后vue2.0也在其核心引入了虚拟DOM的概念。所谓的虚拟DOM实际上就是根据真实的DOM结构创建的与真实的DOM结构一一映射的数据结构。

为什么要做模拟DOM结构这种事呢？当需要对某个DOM节点进行修改的时候，如果是用传统的源生api或jQuery去操作DOM，浏览器大概要经过这样一系列的流程：创建DOM tree –> 创建Style Rules -> 构建Render tree -> 布局Layout –> 绘制Painting。每当更新一次DOM节点，浏览器就会从构建DOM树开始从头到尾执行一遍流程。因此对真实DOM的操作的代价是很昂贵的，一旦DOM有很多结点，DOM操作又很频繁时，就会大大影响浏览器的性能，还会让代码变得难以维护。

下面就来看一个简单的例子，使用JS对象来模拟一个真实的DOM节点。可以通过对这个JS对象进行遍历还原出原有的DOM结构，从而达到在渲染页面时只需要通过操作虚拟DOM就能完成的效果：

# 虚拟DOM的具体实现

用JS对象模拟DOM节点的好处是，页面的更新可以先全部反映在JS对象上，等到更新完后，再将最终的JS对象映射成真实的DOM，交由浏览器去绘制。这样就避免了直接操作真实DOM印发的重排重绘，也提高了性能。这一节就来讲将刚才创建的虚拟DOM还原为真实DOM具体操作。

function vNode(tag, children, text) {
  this.tag = tag
  this.text = text
  this.children = children
}

vNode.prototype.render = function(){
  if(this.tag === '#text') {
    return document.createTextNode(this.text)
  }
  let el = document.createElement(this.tag)
  this.children.forEach(vChild => {
    el.appendChild(vChild.render())
  })
  return el
}

var node = new vNode('div',[],'hello')
node.render()

在上面的例子中，创建了一个虚拟DOM的构造函数，这个构造函数的实例具有一个render()方法。这个方法可以判断虚拟DOM中的节点类型，如果是文本节点，就直接创建文本节点；如果是其他的标签，就先创建该节点，再遍历该节点的子节点，并对这些阶段分别调用render()方法。在用构造函数创建出一个虚拟DOM实例之后，就可以通过调用render()方法实现将整个虚拟DOM树根据它的层级构造出真实的DOM结构。

还可以继续对上面的流程进行优化：

function v(tag, children, text) {
  if(typeof children === 'string') {
    text = children
    children = []
  }
  return new VNode(tag, children, text)
}

# 虚拟DOM的更新

要想做到虚拟DOM的部分更新，就需要区分出前后两个虚拟DOM结构中数据更新的内容，但是要如何获取到前后两次更新的内容呢？这里就要用到diff算法了，在课上并没有对diff算法过多的讲解，因此这里也不在此深入研究，只简单地讲一下。

虚拟DOM中应用diff算法是为了找出需要更新的节点，找出本次DOM必须更新的节点来更新，其他的不更新。整个diff算法的操作流程大致如下：

  1. 首次渲染：patch(node，vnode)，vnode节点会和真实dom节点做关联，同时把外层容器替代掉

  2. 更新 patch(vnode，newVnode)，遍历vnode，通过diff算法对比，相同则递归对比，不同则通过关联找到并修改真实节点；

下面就用课上的例子简单地示范这个过程：

function patchElement(parent, newVNode, oldVNode, index = 0) {
  if(!oldVNode) {     //如果不存在更新前的虚拟DOM，即初次渲染
    parent.appendChild(newVNode.render())
  } else if(!newVNode) {    //如果更新后的节点不存在，即删除节点
    parent.removeChild(parent.childNodes[index])
  } else if(newVNode.tag !== oldVNode.tag || newVNode.text !== oldVNode.text) {//新旧标签或文本不相等
    parent.replaceChild(newVNode.render(), parent.childNodes[index])  //将旧节点修改为新节点
  } else {    // 对新旧虚拟DOM的子节点进行遍历，并进行同样的比较
    for(let i = 0; i < newVNode.children.length || i < oldVNode.children.length; i++) {
      patchElement(parent.childNodes[index], newVNode.children[i], oldVNode.children[i], i)
    }
  }
}

其中parent为需要在页面中关联的父元素，newNode为更新后的虚拟DOM，oldNode为更新前的虚拟DOM。上面的函数大致模拟了diff算法的流程，对于diff算法的具体实现我也没有多少了解，以后有机会再做深入的探讨了。就上面的例子来看，节点的更新涉及到下面三种情况：

  1. patchElement(root , vNode1) 新增节点
  2. patchElement(root , null , vNode1) 删除节点
  3. patchElement(root , vNode1 , vNode2) 修改节点

因此上面只是一个比较简单的比较方法，如果调换节点的位置，这个函数是检测不出来的。不过这个函数已经基本实现了虚拟DOM发现更新内容并给旧内容打补丁的思想。

今天关于虚拟DOM的内容基本就差不多了，只是大致的把虚拟DOM的原理整理了一下，没有深入地去探讨。估计以后接触Vue及React的时候可能会再做一些归纳。最后用一个简单的总结来结束这次博客吧，这也是本篇博客大概内容的一个概括：

为什么要用虚拟DOM？

当修改真正的DOM树的时候，因为DOM中元素节点有许多的属性和方法，当DOM中节点过多时往往需要消耗很大的性能。

虚拟DOM的实现？

使用js对象来表示DOM树的信息和结构，这个js对象可以构建一个真正的DOM树。当状态变更的时候用修改后的新渲染的的js对象和旧的虚拟DOM js对象作对比，获取两棵树的差异，并把差异反映到真实的DOM 结构上。最后操作真正的DOM的时候只操作有差异的部分就可以了。
