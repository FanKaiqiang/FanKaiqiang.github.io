【记录】HTTP入门学习

广宇 2018-9-8 HTTP

这是我的第三篇博客。和之前的两篇一样，还是我关于学习的一些记录。这篇博客我就来介绍一下HTTP和在课上讲过的关于HTTP的一些知识点。

以下摘自百度百科：

超文本传输协议（HTTP，HyperText Transfer Protocol)是互联网上应用最为广泛的一种网络协议。所有的WWW文件都必须遵守这个标准。它可以使浏览器更加高效，使网络传输减少。它不仅保证计算机正确快速地传输超文本文档，还确定传输文档中的哪一部分，以及哪部分内容首先显示(如文本先于图形)等。

HTTP协议采用了请求/响应模型。客户端向服务器发送一个请求，请求头包含请求的方法、URL、协议版本、以及包含请求修饰符、客户信息和内容的类似于MIME的消息结构。服务器以一个状态行作为响应，响应的内容包括消息协议的版本，成功或者错误编码加上包含服务器信息、实体元信息以及可能的实体内容.

# HTTP的请求与响应

# 1、请求

通常HTTP消息包括客户机向服务器的请求消息和服务器向客户机的响应消息。请求消息的格式一般如下，分为四个部分，第4部分可以为空，所以至少包含三个部分：

1 动词 路径 协议/版本
2 Key1: value1
2 Key2: value2
2 Key3: value3
2 Content-Type: application/x-www-form-urlencoded
2 Host: www.baidu.com
2 User-Agent: curl/7.54.0
3 
4 要上传的数据

  • 第1部分为请求行，用来说明请求类型,要访问的资源以及所使用的HTTP版本。动词代表对请求的操作，一般有GET（获取）、POST（新增）、PUT（更新）、PATCH（部分更新）、DELETE（删除）、HEAD（只请求页面的首部）、OPTIONS（允许客户端查看服务器的性能）等。路径为要访问资源的路径，这里的路径包括查询参数，但不包括锚点，如果没有写路径，则路径默认为 /。

  • 第2部分为请求头部，紧接着请求行之后的部分，用来说明服务器要使用的附加信息。 Content-Type 标注了第4部分的格式。HOST指出了请求的目的地。User-Agent,服务器端和客户端脚本都能访问它,它是浏览器类型检测逻辑的重要基础。

  • 第3部分为请求头部后面必须的回车，即使第4部分的请求数据为空，也必须有空行。

  • 第4部分为请求数据，也叫主体，可以添加任意的其他数据。

# 2、响应

响应消息的格式一般如下，和请求部分一样，也是分为四个部分：

1 协议/版本号 状态码 状态解释
2 Key1: value1
2 Key2: value2
2 Content-Length: 17931
2 Content-Type: text/html
3
4 要下载的内容

  • 第1部分为状态行，由HTTP协议版本号，状态码，状态消息三部分组成。状态码是服务器对浏览器说的话，有三位数字组成，第一个数字定义了响应的类别，共分五种类别:

    • 1xx：指示信息--表示请求已接收，继续处理
    • 2xx：成功--表示请求已被成功接收、理解、接受
    • 3xx：重定向--要完成请求必须进行更进一步的操作
    • 4xx：客户端错误--请求有语法错误或请求无法实现
    • 5xx：服务器端错误--服务器未能实现合法的请求

  • 第2部分为消息报头，用来说明客户端要使用的一些附加信息。Content-Type 标注了第4部分的格式，并且遵循MIME规范。

  • 第3部分为消息报头后面必须的回车，即使第4部分的请求数据为空，也必须有空行。

  • 第4部分为服务器返回给客户端的文本信息。

# 3、用Chrome开发者工具查看 HTTP 请求与响应

前文了解了请求和响应消息，接下来将通过在Chrome上查看HTTP的请求和响应信息，更深入地理解HTTP协议的实际应用。查看请求或响应的步骤分别如下：

  1. 打开 Network
  2. 地址栏输入网址
  3. 在 Network 点击，查看 request，点击「view source」就可以看到请求的前三部分了
  4. 查看 Response Headers，点击「view source」你会看到响应的前两部分
  5. 如果有请求的第四部分，那么在 FormData 或 Payload 里面可以看到
  6. 查看 Response 或者 Preview，你会看到响应的第4部分

# 如何使用 curl 命令

本节就来谈谈如何使用curl命令。curl是利用URL语法在命令行方式下工作的开源文件传输工具。curl支持包括HTTP、HTTPS、ftp等众多协议，还支持POST、cookies、认证、从指定偏移处下载部分文件、用户代理字符串、限速、文件大小、进度条等特征。由此可见curl命令的功能是比较强大的。 下面是curl命令的基本语法：

curl(选项)(url)

其中的【选项】为curl的命令参数，有很多，下面列出几个常见的参数：

选项 功能
-A/--user-agent <string> 设置用户代理发送给服务器
-b/--cookie <name=string/file> cookie字符串或文件读取位置
-c/--cookie-jar <file> 操作结束后把cookie写入到这个文件中
-C/--continue-at <offset> 断点续转
-D/--dump-header <file> 把header信息写入到该文件中
-e/--referer 来源网址
-f/--fail 连接失败时不显示http错误
-o/--output 把输出写到该文件中
-O/--remote-name 把输出写到该文件中，保留远程文件的文件名
-r/--range <range> 检索来自HTTP/1.1或FTP服务器字节范围
-s/--silent 沉默模式。不输出任何东西
-T/--upload-file <file> 上传文件
-u/--user <user[:password]> 设置服务器的用户和密码
-w/--write-out [format] 什么输出完成后
-x/--proxy <host[:port]> 在给定的端口上使用HTTP代理
-#/--progress-bar 进度条显示当前的传送状态

# HTTP 状态码

时隔四个月回来重新整理了一下这篇博客，现在看来真是青涩得很，连 Markdown 语法都不怎么会，虽然想把博客弄好看点，可是还是弄得特别丑。这次补充主要是为了添加一些关于 HTTP 状态码的内容。HTTP状态码（英语：HTTP Status Code）是用以表示网页服务器超文本传输协议响应状态的3位数字代码，所有状态码的第一个数字代表了响应的五种状态之一。状态码的数量比较多，这里列出几种比较重要的，以俟面试之需：

  • 状态码200：表示请求已成功，请求所希望的响应头或数据体将随此响应返回
  • 状态码201：表示请求成功并且服务器创建了新的资源，且其 URI 已经随Location 头信息返回。假如需要的资源无法及时建立的话，应当返回 '202 Accepted'。
  • 状态码202：表示服务器已接受请求，但尚未处理。正如它可能被拒绝一样，最终该请求可能会也可能不会被执行。
  • 状态码301：被请求的资源已永久移动到新位置。服务器返回此响应（对 GET 或 HEAD 请求的响应）时，会自动将请求者转到新位置。
  • 状态码302：请求的资源临时从不同的URI响应请求，但请求者应继续使用原有位置来进行以后的请求。
  • 状态码304：未修改。所请求的资源未修改，服务器返回此状态码时，不会返回任何资源。客户端通常会缓存访问过的资源，通过提供一个头信息指出客户端希望只返回在指定日期之后修改的资源
  • 状态码401：请求要求身份验证。 对于需要登录的网页，服务器可能返回此响应。
  • 状态码403：服务器已经理解请求，但是拒绝执行它。
  • 状态码404：请求失败，请求所希望得到的资源未被在服务器上发现。
  • 状态码500：服务器遇到了一个未曾预料的状况，导致了它无法完成对请求的处理。一般来说，这个问题都会在服务器的程序码出错时出现。
  • 状态码503：由于临时的服务器维护或者过载，服务器当前无法处理请求。
